# 논문 분석 에이전트 구상

## 목표
기존의 "PDF 업로드 후 1회 분석" 흐름을, 사용자의 목적에 따라 다음 행동까지 이어주는 에이전트형 분석기로 확장한다.

## 권장 동작 흐름
1. 사용자가 PDF를 업로드한다.
2. 에이전트가 논문 분야, 길이, 텍스트 품질을 먼저 점검한다.
3. 에이전트가 적절한 모델과 분석 전략을 선택한다.
4. 기본 분석 결과를 생성한다.
5. 결과를 바탕으로 후속 액션을 제안한다.
6. 사용자가 원하는 작업을 추가로 요청하면 이어서 수행한다.

## 에이전트가 대신 해줄 수 있는 일
- 논문 구조 요약
- 핵심 변수와 분석 기법 정리
- 연구 질문과 결론 추출
- 연구 한계와 후속 연구 제안
- 여러 논문 간 비교 분석
- Research Gap 후보 제안
- 발표 자료용 짧은 요약 재작성
- 지도교수 보고용 포맷 변환

## 현재 앱에 붙이기 좋은 방식
### 1. 단일 분석 에이전트
- 업로드 직후 한 번의 요청으로 구조화된 JSON을 만든다.
- 현재 구현과 가장 가깝고 빠르게 적용 가능하다.

### 2. 단계형 워크플로 에이전트
- `parser`
- `analyzer`
- `gap-finder`
- `writer`
- 각 단계를 분리해 재시도와 품질 보정을 쉽게 만든다.

### 3. 대화형 에이전트
- 분석이 끝난 뒤 사용자가
- "이 논문의 한계만 더 자세히"
- "발표용 5줄 요약으로 바꿔줘"
- "A 논문과 B 논문 차이만 보여줘"
- 같은 후속 요청을 이어서 할 수 있게 만든다.

## 추천 구현 순서
1. 현재 `/api/parse-pdf` 결과를 만드는 분석 단계를 `agent.run()` 형태로 감싼다.
2. 입력으로 `goal`, `tone`, `outputFormat`을 추가한다.
3. 저장된 논문을 다시 불러와 후속 질문에 답하는 `/api/agent-chat`을 만든다.
4. 비교 분석과 Gap 분석도 같은 에이전트에서 호출하도록 통합한다.

## 범용 API 구조 아이디어
- `provider`: `openai | gemini | claude`
- `model`: 실제 모델명
- `apiKeySource`: `server_env | user_supplied`
- `mode`: `analyze | compare | gap | rewrite`

이렇게 두면 같은 UI에서 공급자만 바꿔 재사용할 수 있다.

## 운영 관점 추천
- 첫 배포는 `BYOK(사용자 키 입력)` 방식이 가장 현실적이다.
- 서버 운영용 키는 관리자 전용 환경변수로 분리한다.
- 체험 모드는 mock 응답 또는 샘플 PDF로 제공한다.
- 분석 실패 시 다른 모델로 자동 fallback 하는 옵션을 둔다.
